**Date:** 2026-03-18
**Status:** Approved

## 1. Purpose

ProdMaster AI is a standalone Claude Code plugin that layers a meta-orchestration, measurement, and intelligence system on top of Superpowers. Where Superpowers owns the code execution cycle (TDD → implement → review), ProdMaster AI owns the layer above: what to build next, is it working, what did we learn, and what should change — including the plugin itself.

The defining capability is **auto-evolution**: the plugin improves and extends itself over time using three learning signals (outcomes, research, user feedback), and can contribute those improvements back to its own upstream codebase for all users.

---

## 2. Architecture: Three Layers

### Layer 1 — Hooks (passive, always-on)
A `SessionStart` hook reads the `memory/` layer and injects current project state, learned patterns, and active priorities into every Claude Code conversation automatically.

**Hook execution contract:**
- `hooks/hooks.json` registers a `SessionStart` event that runs the platform hook runner on every session start
- On Windows: `run-hook.cmd session-start`. On Unix: `run-hook.sh session-start`
- **Cross-platform dispatch:** A single `hooks.json` entry uses a universal wrapper. See Section 5 for the hooks.json definition and platform detection approach.
- The runner reads: `memory/project-context.md` (full), `memory/patterns.md` (last 5 entries), `memory/skill-gaps.md` (open gaps only), `memory/evolution-log.md` (last 3 entries)
- It populates `hooks/session-start.md` template placeholders: `{{active_features}}`, `{{top_patterns}}`, `{{open_gaps}}`, `{{recent_evolutions}}`
- Writes the rendered output to stdout
- Claude Code captures stdout from `SessionStart` hooks and prepends it to the system prompt context
- `session-start.md` is a prompt template only — it is never executed

### Layer 2 — Skills (triggered by task type)
Six focused markdown skill files. Each has a single responsibility. Skills read from and write to the memory layer. `evolve-self` is the only skill that modifies other skill files and generates new ones.

### Layer 3 — Memory System (the living brain)
Structured markdown files in `memory/`. Every skill appends dated entries — nothing is overwritten. Grows every cycle.

## 3. Skill File Format

Every `SKILL.md` uses this frontmatter. Required for all existing skills and all skills generated by `evolve-self`:

```yaml
---
name: skill-name
description: One-line trigger description (Claude Code uses this to auto-suggest the skill)
version: 1.0.0
triggers:
  - plain-language condition that causes this skill to activate
reads:
  - memory/file.md
writes:
  - memory/file.md
generated: false        # true if created by evolve-self
generated_from: ""      # skill-gaps.md entry id that triggered generation (if generated: true)
---
```

## 4. Plugin Manifest Schema

`/.claude-plugin/plugin.json`:

```json
{
  "name": "prodmaster-ai",
  "version": "1.0.0",
  "description": "Meta-orchestration, measurement, and self-evolving intelligence layer for Claude Code.",
  "category": "productivity",
  "author": {
    "name": "",
    "email": ""
  },
  "upstream": {
    "repo": "",
    "branch": "main",
    "pr_label": "auto-evolved"
  }
}
```

The `upstream` block is optional. If absent or if `repo` is empty, `evolve-self` runs in **local-only mode** — all evolution stays local, no PRs are created. If present, `repo` must be a valid `owner/repo` GitHub slug. Auth delegation order: GitHub MCP server (if registered in Claude Code) → `gh` CLI (if available) → local-only fallback with user notification.

## 5. Hooks Configuration

**`hooks/hooks.json`** — cross-platform via a single universal wrapper entry:

```json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "\"${CLAUDE_PLUGIN_ROOT}/hooks/run-hook.cmd\" session-start",
            "async": false
          }
        ]
      }
    ]
  }
}
```

**Cross-platform strategy:** `run-hook.cmd` is a Windows batch file that:
1. Detects the OS (Windows always runs `.cmd`, so this is the Windows entry point)
2. On the first line, it calls `run-hook.sh` via WSL if running under WSL, otherwise runs the Windows batch logic directly

On Unix systems (macOS, Linux), users replace the `run-hook.cmd` reference with `run-hook.sh` in `hooks.json` post-install, or the install script auto-patches this. The install documentation in `docs/README.md` specifies this step.

`run-hook.cmd` and `run-hook.sh` both:
- Receive the hook event name as `$1` / `%1` (`session-start`)
- Read the memory files listed in Section 2
- Fill `session-start.md` template placeholders
- Write rendered output to stdout
- Exit with code 0 on success, 1 on error (Claude Code ignores hook failures gracefully)

## 6. Memory Files — Schemas and Initial State

All memory files ship with pre-seeded initial content. This ensures the first run of any skill has a valid structure to read and append to.

### 6.1 `memory/project-context.md`

Read/written by: `orchestrate`, `measure` (counter only), `decide`, SessionStart hook.

**Full section schema:**

```markdown
---
total_tasks_completed: 0
last_evolved_at_task: 0
evolve_every_n_tasks: 10
---

## Active Features
<!-- orchestrate appends entries here: "- YYYY-MM-DD: <feature name> [status: active|blocked|done]" -->

## Blockers
<!-- orchestrate/decide append entries: "- YYYY-MM-DD: <blocker description> | age_days: <n> | recommended_fix: <text>" -->

## Decisions Log
<!-- decide appends entries:
---
date: YYYY-MM-DD
decision: <what was decided>
rationale: <why>
options_considered: [A, B, C]
status: pending_outcome | confirmed_good | confirmed_bad
---
-->
```

The `total_tasks_completed` and `last_evolved_at_task` fields live in the YAML frontmatter of this file. `measure` reads the current value, increments it, and rewrites only the frontmatter block. `evolve-self` compares `total_tasks_completed` against `last_evolved_at_task + evolve_every_n_tasks` to determine if it should run.

### 6.2 `memory/skill-performance.md`

Written by: `measure`. Read by: `report`, `decide`, `evolve-self`.

Each entry is a YAML block separated by `---`:

```yaml
---
date: YYYY-MM-DD
feature: <feature name>
tasks_completed: <n>
velocity_tasks_per_week: <n>
qa_pass_rate: <0.0-1.0>
review_iterations: <n>
time_per_feature_hours: <n>
blockers: <n>
blocker_age_days_avg: <n>
---
```

Initial state: file contains a header comment and one example entry marked `example: true` so `evolve-self` and `report` can detect an empty-data state.

### 6.3 `memory/skill-gaps.md`

Written by: `learn`. Read by: `evolve-self`.

Each gap entry:

```markdown
---
id: gap-YYYY-MM-DD-<slug>
pattern: <plain description of the unhandled pattern>
first_seen: YYYY-MM-DD
last_seen: YYYY-MM-DD
occurrences: <n>
status: open | generating | generated
generated_skill: ""   # filled when status = generated
---
```

`evolve-self` Mode 2 reads entries where `occurrences >= 3` and `status: open`.

### 6.4 `memory/patterns.md`

Written by: `learn` (auto path). Read by: `evolve-self`, SessionStart hook, `report`.

Each entry:
```markdown
---
date: YYYY-MM-DD
pattern: <description of what worked>
context: <what kind of task/feature this applied to>
qa_pass_rate: <from the cycle>
review_iterations: <from the cycle>
---
```

### 6.5 `memory/mistakes.md`

Written by: `learn` (auto path). Read by: `evolve-self`, `report`.

Each entry:
```markdown
---
date: YYYY-MM-DD
mistake: <what went wrong>
root_cause: <why it went wrong>
fix_applied: <what resolved it>
qa_pass_rate: <from the cycle>
review_iterations: <from the cycle>
---
```

### 6.6 `memory/feedback.md`

Written by: `learn` (feedback path **only**). Read by: `evolve-self` (read-only — never auto-PR'd without user gate).

Each entry:
```markdown
---
date: YYYY-MM-DD
feedback: <verbatim user input>
context: <what was happening when user gave feedback>
contributed_upstream: false   # set to true after user-approved PR
---
```

### 6.7 `memory/research-findings.md`

Written by: `evolve-self` (via research subagent). Read by: `evolve-self`.

Each entry:
```markdown
---
date: YYYY-MM-DD
research_question: <what the subagent was asked to investigate>
skill_target: <which skill this applies to>
finding: <what the research found>
source: <where the finding came from — URL, doc name, or "Context7">
confidence: high | medium | low
applied: false   # set to true when a skill rewrite uses this finding
---
```

### 6.8 `memory/evolution-log.md`

Written by: `evolve-self` (local changes only). Never auto-PR'd.

Each entry:
```markdown
---
date: YYYY-MM-DD
mode: improve | generate
skill: <skill name>
trigger: <what data pattern triggered this>
change_summary: <one sentence>
---
```

## 7. Skills

### 7.1 `orchestrate`
**Triggers:** User states a high-level feature goal ("Build X", "Start work on Y")
**Reads:** `memory/project-context.md`, `memory/patterns.md`
**Writes:** `memory/project-context.md`
**Job:** Breaks the goal into Superpowers-compatible task cycles. Tracks cross-feature dependencies. Decides what to work on next based on blockers, ROI, and current project context. Invokes Superpowers workflows — does not reimplement them.

### 7.2 `measure`
**Triggers:** After every completed Superpowers cycle
**Reads:** `memory/project-context.md` (to increment `total_tasks_completed`)
**Writes:** `memory/skill-performance.md` (new YAML entry), `memory/project-context.md` (frontmatter counter only)
**Job:** Appends a dated YAML entry (schema in Section 6.2). Increments `total_tasks_completed` in `project-context.md` frontmatter.

### 7.3 `report`
**Triggers:** On demand (`/report`) OR weekly (see weekly scheduling note below)
**Reads:** `memory/skill-performance.md`, `memory/project-context.md`, `memory/patterns.md`, `memory/mistakes.md`
**Writes:** `reports/weekly-YYYY-MM-DD.md`, `reports/dashboard.html`
**Job:** Generates two outputs. See Section 11 for dashboard panel → source mapping.

**Weekly scheduling:** Claude Code does not have a native cron scheduler built into plugins. Weekly reports are triggered one of two ways: (a) user runs `/report weekly`, or (b) if the `anthropic-skills:schedule` plugin is installed, a scheduled task entry can be configured to call `report` on a weekly cadence. The `docs/README.md` documents both options.

### 7.4 `decide`
**Triggers:** User at a decision fork ("should we do A or B?", "what should we prioritise?")
**Reads:** `memory/project-context.md`, `memory/skill-performance.md`
**Writes:** `memory/project-context.md` (`## Decisions Log` section)
**Job:** Reads project state and metrics. Ranks options by ROI and risk using velocity/QA data. Returns a clear recommendation with trade-offs. Logs decision with `status: pending_outcome`.

### 7.5 `learn`
**Triggers:** After each Superpowers cycle (auto) AND user explicit feedback (user-gated)
**Reads:** Cycle outcome summary (passed as input by `measure` or user); current skills directory manifest (to detect unhandled patterns — see below)
**Writes:** `memory/patterns.md`, `memory/mistakes.md`, `memory/skill-gaps.md` (auto path); `memory/feedback.md` (feedback path only)

**Detecting unhandled patterns (auto path):** `learn` reads a lightweight pattern manifest at `memory/connectors/skill-pattern-manifest.md` — a list of pattern keywords each skill covers, updated by `evolve-self` whenever a skill is created or rewritten. If the cycle contains a recurring pattern keyword not in the manifest, `learn` appends it to `skill-gaps.md`.

**Auto path thresholds:**
- `qa_pass_rate >= 0.9` AND `review_iterations <= 2` → write to `patterns.md`
- `qa_pass_rate < 0.6` OR `review_iterations > 4` → write to `mistakes.md`
- Unhandled pattern detected → append to `skill-gaps.md` (or increment `occurrences` if entry already exists)

**Feedback path:** User provides explicit input. `learn` writes verbatim to `memory/feedback.md`. No other automated process writes to this file.

### 7.6 `evolve-self`
**Triggers:** `total_tasks_completed` in `project-context.md` frontmatter crosses `last_evolved_at_task + evolve_every_n_tasks` OR user runs `/evolve`
**Reads:** All memory files, `skills/` directory
**Writes:** `skills/*/SKILL.md` (rewrites), `skills/<new>/SKILL.md` (new files), `memory/research-findings.md`, `memory/evolution-log.md`, `memory/pending-upstream/<slug>.md`, `memory/pending-upstream/last-pr.txt`, `EVOLUTION-LOG.md` (root), `memory/connectors/skill-pattern-manifest.md`

**Execution order and empty-result behaviour:**
1. Mode 1 (Improve) runs first
2. Mode 2 (Generate) runs second
3. If neither mode finds anything actionable → writes a dated "no-op" entry to `memory/evolution-log.md` and notifies user: *"Evolution check ran — no improvements or new skills needed at this time."*
4. Both modes can produce output in the same invocation

**Mode 1 — Improve existing skills:**
1. Reads `skill-performance.md`. Identifies skills with declining metrics (success rate declining across last 5 entries with `qa_pass_rate` trend).
2. Dispatches research subagent via `skills/evolve-self/research-subagent-prompt.md`. See Section 8 for the research subagent contract.
3. Research subagent writes findings to `memory/research-findings.md`.
4. If `confidence: high` or `confidence: medium` finding exists → rewrites target `SKILL.md` locally.
5. Appends entry to `memory/evolution-log.md`.
6. For upstream: writes proposal to `memory/pending-upstream/<slug>.md` (Section 9 schema). Sets `occurrence_count: 1` (Mode 1 improvements are triggered by metric trends, not occurrence counts; the `>= 3` gate applies to Mode 2 only). Mode 1 proposals go through the same research-validation step before auto-PR.

**Mode 2 — Generate new skills:**
1. Reads `memory/skill-gaps.md` for entries where `occurrences >= 3` AND `status: open`.
2. Creates `skills/<name>/SKILL.md` with full frontmatter (`generated: true`, `generated_from: <gap entry id>`).
3. Updates `memory/connectors/skill-pattern-manifest.md` with the new skill's pattern keywords.
4. Sets gap entry `status: generating` then `status: generated` after file is written.
5. Appends to `memory/evolution-log.md`.
6. Plugin auto-picks up new skill files on next session start.

**Upstream pipeline:**
1. Packages change as `memory/pending-upstream/<slug>.md` (Section 9 schema).
2. Research subagent validates against current plugin codebase (no duplicate proposals).
3. Decision:
   - **Outcome/research-sourced** → auto-PR to `plugin.json upstream.repo`. Branch: `auto-evolved/YYYY-MM-DD-<slug>`. Body: `skills/evolve-self/pr-template.md`. Label: `auto-evolved`.
   - **Feedback-sourced** → pauses and asks user: *"I've prepared an improvement based on your feedback. Want to contribute this back to the plugin for all users?"* Creates PR only on explicit approval.
4. Rate limit: checks `memory/pending-upstream/last-pr.txt`. Format: `YYYY-MM-DDTHH:MM:SSZ` (ISO 8601 UTC). If fewer than 24 hours since last PR, queues the proposal and notifies user.
5. PRs never self-merge.
6. **Merge status check:** On every `evolve-self` run, for each proposal file in `memory/pending-upstream/` with `status: pr_created`, queries the GitHub API (via GitHub MCP or `gh` CLI) to check merge status. If merged → updates proposal `status: pr_merged`, appends entry to root `EVOLUTION-LOG.md`, updates `memory/evolution-log.md`. If closed without merge → sets `status: rejected`.
7. If GitHub unavailable: logs proposal locally, writes human-readable summary to stdout, notifies user to contribute manually.

**After any Mode 1 or Mode 2 run:** Updates `last_evolved_at_task` in `project-context.md` frontmatter to current `total_tasks_completed`.

## 8. Research Subagent Contract (`skills/evolve-self/research-subagent-prompt.md`)

The research subagent prompt template receives these inputs (filled by `evolve-self` before dispatch):
- `{{skill_name}}` — the target skill
- `{{current_skill_content}}` — current content of the skill file
- `{{performance_data}}` — last 5 entries from `skill-performance.md` for this skill
- `{{research_question}}` — e.g., "Are there better patterns for orchestrating cross-feature dependencies in Claude Code plugins?"

The research subagent must:
1. Use Context7 MCP (if available) to pull current documentation relevant to the question
2. Search for patterns in the existing codebase that may already address the question
3. Return a structured finding in the `memory/research-findings.md` format (Section 6.7)
4. Set `confidence: high` only if it found a concrete, actionable improvement with evidence
5. Set `confidence: low` if findings are speculative or not directly applicable

The research subagent writes directly to `memory/research-findings.md` and returns a summary to `evolve-self`.

## 13. Connector Activation Contract

Each connector file in `memory/connectors/` uses this format:

```yaml
---
connector: github | slack | linear
active: true
---
## Config
mcp_server: <mcp server name as registered in Claude Code>
default_repo: owner/repo      # github only
webhook_url: ""               # slack only
workspace_id: ""              # linear only
## State
last_sync: YYYY-MM-DD
```

Skills check for connector files at runtime. File exists + non-empty + `active: true` → integration path active. File absent or `active: false` → silent file-only fallback.

**Connector behaviour per skill:**

| Connector | Skill | Behaviour when active |
|---|---|---|
| GitHub | `orchestrate` | Reads open GitHub Issues as input for feature breakdown. Writes task cycles back as Issue comments. |
| GitHub | `evolve-self` | Creates upstream PRs (already in core design) |
| Slack | `report` | Posts weekly summary to configured Slack channel via webhook after generating the markdown report |
| Slack | `evolve-self` | Posts a notification when a new skill is generated or a skill is rewritten |
| Linear | `orchestrate` | Reads Linear issues as feature input; updates Linear issue status as task cycles complete |
| Linear | `report` | Includes Linear cycle metrics in the weekly markdown report |

`memory/connectors/skill-pattern-manifest.md` is a special connector file written exclusively by `evolve-self`. It lists pattern keywords per skill for `learn`'s gap detection. It is not a user-configured connector — it is maintained automatically.

## 16. Implementation Clarifications

### 16.1 session-start.md Output Format
`run-hook.cmd` / `run-hook.sh` writes raw markdown to stdout — no wrapping tags. Claude Code prepends raw hook stdout to the system prompt as-is. The `session-start.md` template is plain markdown with `{{placeholder}}` tokens replaced by the hook script. Example rendered output:
```
## ProdMaster AI Context
**Active features:** Login system (active), Dashboard (blocked)
**Top patterns:** Fast delivery with early QA...
**Open gaps:** No skill covers data migration patterns (2 occurrences)
**Recent evolutions:** 2026-03-15: orchestrate skill improved (research-based)
```

### 16.2 Superpowers Integration Surface
ProdMaster AI integrates with Superpowers through **convention, not API**. There is no programmatic interface — integration is via skill invocation and reading shared project files.

**Cycle outcome object** (passed as input to `measure` and `learn`):
```yaml
feature: <feature name>
tasks_completed: <n>
qa_pass_rate: <0.0-1.0>
review_iterations: <n>
time_hours: <n>
blockers_encountered: <n>
patterns_used: [list of pattern keywords]
unhandled_patterns: [list of patterns no skill matched]
```

This object is constructed by the user (or by `orchestrate`) at the end of each Superpowers cycle and passed to `measure` as a prompt input. `orchestrate` invokes Superpowers by triggering `superpowers:writing-plans` and `superpowers:subagent-driven-development` skills via the normal Claude Code skill invocation mechanism — the same way a user would trigger them manually.

### 16.3 skill-gaps.md Occurrence Counting Rules
- One occurrence = one cycle where the unhandled pattern appeared
- If the same pattern appears multiple times within a single cycle, it counts as **one occurrence**
- If the same pattern appears in a subsequent cycle, `learn` finds the existing gap entry (by pattern text match) and increments `occurrences` — it does not create a duplicate entry
- `evolve-self` Mode 2 triggers when `occurrences >= 3` (at least 3, not exactly 3)

### 16.4 research-findings.md `applied` Field
`evolve-self` sets `applied: true` on a `research-findings.md` entry immediately after successfully rewriting the target `SKILL.md`. If the rewrite fails for any reason, `applied` remains `false` and the entry is retried on the next `evolve-self` run.

### 16.5 GitHub API Polling and Failure Handling
- On each `evolve-self` run, it checks all `status: pr_created` proposal files
- Queries GitHub API once per proposal per run (no retry loops during a single run)
- If API call fails: logs a warning to `memory/evolution-log.md` with date and error, leaves `status: pr_created` unchanged, retries on next run
- After 30 days with no merge status change: sets `status: rejected` and logs reason as "timeout — no merge after 30 days"

### 16.6 Rate Limit Queue
When the 24-hour rate limit blocks a PR:
- The proposal file stays in `memory/pending-upstream/` with `status: pending`
- `evolve-self` notifies the user: *"1 upstream proposal is queued — rate limit reached. It will be submitted on the next evolve-self run after [timestamp]."*
- The queue does **not** block Mode 1/2 local improvements — only upstream PR creation is delayed
- On the next `evolve-self` run, queued proposals are submitted first (oldest first) before processing new proposals

### 16.7 Connector Precedence Rules
Multiple connectors can be active simultaneously — they are **additive, not competing**:
- If both GitHub and Linear are active, `orchestrate` reads feature input from **both** and merges them (deduplicating by feature name)
- `report` includes data from all active connectors in the weekly report
- `evolve-self` uses whichever GitHub access method is available for PRs; Slack/Linear do not affect PR creation
- If two connectors produce conflicting feature status for the same item, `project-context.md` entry takes precedence (local state wins)

### 16.8 Skill Name Collision in Mode 2
When `evolve-self` Mode 2 attempts to generate a skill and `skills/<name>/SKILL.md` already exists:
1. **Do not overwrite.** Log a warning to `memory/evolution-log.md`: "Skill generation skipped — skills/<name>/SKILL.md already exists."
2. Set the gap entry `status: generated` with `generated_skill: <existing skill name>`
3. This prevents duplicate work — the existing skill may already cover the gap. If not, the user should improve the existing skill via Mode 1 or `/evolve`.

### 16.9 skill-pattern-manifest.md Schema
```yaml
---
last_updated: YYYY-MM-DD
---
## Skills

### orchestrate
keywords: [feature goal, build, start work, implement, dependency, task cycle]

### measure
keywords: [cycle complete, metrics, velocity, QA pass, review iterations]

### report
keywords: [report, dashboard, weekly summary, status update, management report]

### decide
keywords: [decision, prioritise, which option, trade-off, should we, A or B]

### learn
keywords: [learned, pattern, mistake, feedback, wrong, worked well]

### evolve-self
keywords: [evolve, improve plugin, generate skill, self-improve, /evolve]
```

`evolve-self` appends a new `### <skill-name>` block when it generates a new skill, and updates keywords when it rewrites an existing skill. `learn` uses substring matching against `unhandled_patterns` from the cycle outcome object to detect gaps.

### 16.10 Duplicate PR Validation by Research Subagent
The research subagent checks for duplicates against:
1. All files in `memory/pending-upstream/` (proposals already staged — any status)
2. All existing skill files in `skills/` (if the proposed new skill already exists as a file)
3. The `applied` field in `memory/research-findings.md` (if the finding was already used)

It does **not** check the remote GitHub repository branches — that would require live API access during validation and adds fragility. Remote branch collision is handled by GitHub itself on PR creation (the PR will fail to open against an existing branch, which `evolve-self` logs and retries with a suffixed slug: `<slug>-v2`).

## 17. Key Design Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Architecture | Three-layer: hooks + skills + memory | Only architecture where auto-evolution is first-class |
| Learning signals | Outcomes + research (auto) + feedback (user-gated) | Balances automation with human oversight |
| Evolution threshold | Mode 1: metric trend; Mode 2: 3+ occurrences | Prevents noise; Mode 1 is continuous, Mode 2 is cumulative |
| Upstream safety | Auto-PR for outcomes/research; user-gated for feedback | User's words are opinionated — require consent |
| Upstream PRs | Additive only, never self-merge, rate-limited 1/day | Prevents repo flooding and accidental regressions |
| Upstream fallback | Local-only if GitHub unavailable | Zero hard dependencies on external services |
| Dashboard | Vanilla HTML, no dependencies | Zero setup friction, opens anywhere |
| Connectors | Activation-by-presence with `active: true` flag | No code changes to add/remove integrations |
| Memory writes | Append-only with dates | Preserves full history for trend detection |
| Two evolution logs | `memory/evolution-log.md` (local) + `EVOLUTION-LOG.md` (upstream) | Local history stays internal; upstream log is the public changelog |
| Skill frontmatter | Standard schema with `generated` + `generated_from` | Implementer and evolve-self use the same template |
| Weekly report trigger | `/report weekly` command or `anthropic-skills:schedule` | No native cron in plugin system; documented both options |
| project-context.md counter | YAML frontmatter `total_tasks_completed` field | Single source of truth for evolve-self trigger; read/write by measure |
| Pattern gap detection | `skill-pattern-manifest.md` maintained by evolve-self | Decouples learn from needing to read all skill files directly |
